## 4. 配置说明

打开 `cms_monitor.py`，修改文件顶部 **§ 1 配置区**：

```python
# ── 必填 ──────────────────────────────────────────────────────
API_ENDPOINT      = "https://cms.cn-hangzhou.example.com"  # 云监控 API 地址
ACCESS_KEY_ID     = "YOUR_ACCESS_KEY_ID"                   # AK
ACCESS_KEY_SECRET = "YOUR_ACCESS_KEY_SECRET"               # SK
REGION_ID         = "cn-hangzhou"                          # Region ID

# ── 专有云可选 Header（公共云留空）────────────────────────────
ORG_ID            = ""   # 组织 ID，不填则默认 root 组织
RESOURCE_GROUP_ID = ""   # 资源集 ID
INSTANCE_ID       = ""   # 实例 ID（列表查询时留空）
```

> **如何获取 API_ENDPOINT？**
> 登录 Apsara Uni-manager 运营控制台 → 产品 → 其他 → API与工具 → API目录，
> 搜索 `DescribeMetricLast` 即可查看当前环境的 Endpoint 地址。

> **如何获取 REGION_ID？**
> 控制台页面 URL 中通常包含 regionId 参数，或联系管理员获取。

> 所有资源的最新数据点，脚本从中提取维度字段并去重，即得到资源列表。
> 注意：**只有近期有监控数据的资源**才会出现在列表中。

### 6.4 OSS 监控采集

OSS 共有 **54 个监控指标**，分为 4 个类别。

#### 采集全部指标（最新值）

```bash
python cms_monitor.py oss-metrics --bucket my-bucket --uid 123456789
```

#### 按类别采集（推荐，避免一次请求过多）

```bash
# 服务总览（15个）：可用性、请求量、流量、存储大小等
python cms_monitor.py oss-metrics --bucket my-bucket --uid 123456789 --category overview

# 请求状态详情（16个）：各类错误请求数/占比、成功/重定向请求数
python cms_monitor.py oss-metrics --bucket my-bucket --uid 123456789 --category request

# 最大延时（14个）：各操作的 E2E 延时和服务器延时
python cms_monitor.py oss-metrics --bucket my-bucket --uid 123456789 --category latency

# 成功请求操作分类（9个）：各操作（Get/Put/Delete等）成功请求数
python cms_monitor.py oss-metrics --bucket my-bucket --uid 123456789 --category ops
```

#### 采集指定指标

```bash
python cms_monitor.py oss-metrics \
  --bucket my-bucket \
  --uid 123456789 \
  --metrics Availability,TotalRequestCount,InternetSend
```

#### 采集历史时序数据

```bash
# 默认查询过去 1 小时
python cms_monitor.py oss-history \
  --bucket my-bucket \
  --uid 123456789 \
  --metric Availability

# 指定时间范围（Unix 时间戳，单位：秒）
python cms_monitor.py oss-history \
  --bucket my-bucket \
  --uid 123456789 \
  --metric TotalRequestCount \
  --start 1741104000 \
  --end   1741107600 \
  --period 60

# 指定时间范围（可读格式）
python cms_monitor.py oss-history \
  --bucket my-bucket \
  --uid 123456789 \
  --metric InternetSend \
  --start "2026-03-05 10:00:00" \
  --end   "2026-03-05 11:00:00"
```

输出示例：
```
指标: Availability，共 60 个数据点
  [2026-03-05 10:00:00]  Average=100.0  Maximum=100.0  Minimum=100.0
  [2026-03-05 10:01:00]  Average=100.0  Maximum=100.0  Minimum=100.0
  ...
```

### 6.5 MQ 监控采集

MQ 有 4 个维度层级，每层维度对应不同的指标集。

#### 实例级（维度：userId + instanceId）

```bash
# 采集全部实例级指标（5个）
python cms_monitor.py mq-instance \
  --instance MQ_INST_aaa111 \
  --uid 123456789

# 采集指定指标
python cms_monitor.py mq-instance \
  --instance MQ_INST_aaa111 \
  --uid 123456789 \
  --metrics SendMessageCountPerInstance,ReceiveMessageCountPerInstance
```

#### GroupID 级（维度：userId + instanceId + groupId）

重点关注 **ConsumerLag（消息堆积量）**，堆积过大说明消费者处理能力不足。

```bash
python cms_monitor.py mq-group \
  --instance MQ_INST_aaa111 \
  --group    GID_order_consumer \
  --uid      123456789

# 只看消息堆积量
python cms_monitor.py mq-group \
  --instance MQ_INST_aaa111 \
  --group    GID_order_consumer \
  --uid      123456789 \
  --metrics  ConsumerLag
```

#### Topic 级（维度：userId + instanceId + topic）

```bash
python cms_monitor.py mq-topic \
  --instance MQ_INST_aaa111 \
  --topic    order_topic \
  --uid      123456789
```

#### GroupID+Topic 级（维度：userId + instanceId + groupId + topic）

精细到某个消费者组消费某个 Topic 的堆积情况。

```bash
python cms_monitor.py mq-gid-topic \
  --instance MQ_INST_aaa111 \
  --group    GID_order_consumer \
  --topic    order_topic \
  --uid      123456789
```

#### MQ 历史时序数据

```bash
# 查看某实例消息堆积量过去 1 小时的变化
python cms_monitor.py mq-history \
  --instance MQ_INST_aaa111 \
  --uid      123456789 \
  --metric   ConsumerLag \
  --group    GID_order_consumer

# 指定时间范围
python cms_monitor.py mq-history \
  --instance MQ_INST_aaa111 \
  --uid      123456789 \
  --metric   SendMessageCountPerInstance \
  --start    "2026-03-05 09:00:00" \
  --end      "2026-03-05 10:00:00"

# GroupID+Topic 组合维度的历史数据
python cms_monitor.py mq-history \
  --instance MQ_INST_aaa111 \
  --uid      123456789 \
  --metric   ConsumerLagPerGidTopic \
  --group    GID_order_consumer \
  --topic    order_topic
```

## 9. 常见问题排查

### 问题 1：`test` 命令报 `HTTP 403` 或签名错误

```
RuntimeError: HTTP 403: {"Code":"InvalidAuthorization","Message":"..."}
```

**排查步骤：**
1. 检查 `ACCESS_KEY_ID` 和 `ACCESS_KEY_SECRET` 是否填写正确
2. 确认系统时间准确（签名中含时间戳，偏差超过 15 分钟会拒绝）：
   ```bash
   date -u   # 查看 UTC 时间
   ```
3. 开启 `--debug` 查看实际发送的请求 URL

---

### 问题 2：`test` 命令报连接超时或拒绝

```
urllib.error.URLError: <urlopen error [Errno 111] Connection refused>
```

**排查步骤：**
1. 确认 `API_ENDPOINT` 地址可达：
   ```bash
   curl -k https://cms.cn-hangzhou.example.com/
   ping cms.cn-hangzhou.example.com
   ```
2. 检查防火墙/安全组是否放通 443 端口
3. 专有云环境确认 HTTPS 证书或改用 HTTP 地址

---

### 问题 3：`list-mq-instances` 返回 0 个实例

**原因**：资源发现依赖监控数据，**近期没有消息收发的实例不会出现**。

**解决方案**：
- 先在控制台确认实例存在且有消息收发
- 如果实例确实无流量，直接手动指定 `--instance` 参数采集

---

### 问题 4：`oss-metrics` 全部返回"暂无数据"

**原因**：
- Bucket 名称大小写错误（`--bucket` 参数区分大小写）
- `--uid` 与实际账号 userId 不符
- 该 Bucket 近期无访问，部分指标无数据点（正常现象）

**验证方式**：
```bash
# 先列出实际存在数据的 Bucket
python cms_monitor.py list-oss-buckets

# 再用返回的 BucketName 和 userId 采集
python cms_monitor.py oss-metrics --bucket <实际名称> --uid <实际userId>
```

### 问题 5：SSL 证书错误

**专有云自签证书场景**，脚本已默认关闭 SSL 校验（`verify_mode = CERT_NONE`），
通常不会出现证书错误。如果仍报错，确认 `API_ENDPOINT` 使用的是 `https://` 还是 `http://`。
